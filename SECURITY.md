# Security policy

## Supported versions

Security fixes are applied to the latest commit on the default branch (`main` or `master`). There are no separate long-term support branches unless the maintainers announce otherwise.

## Reporting a vulnerability

**Please do not** open a public GitHub issue that describes a security vulnerability or includes proof-of-concept exploit code.

Instead, contact the repository maintainers privately:

- Prefer **GitHub Security Advisories** for this repository (**Security** tab → **Report a vulnerability**), if enabled.
- If that is not available, email or message the maintainers through the contact channels listed in the repository or organization profile.

Include:

- A short description of the issue and its impact
- Steps to reproduce (minimal reproduction is ideal)
- Any relevant versions (Node, OS, dependency versions if applicable)

You should receive an acknowledgment within a reasonable time; please allow maintainers time to assess and ship a fix before public disclosure.

## Disclosure

We ask that you **coordinate disclosure** with maintainers (typically 90 days or less after a fix is available, depending on severity). Critical issues may be patched and disclosed sooner.

## Scope

This policy covers this repository and the application it ships. Third-party services (hosting, LLM providers, databases you connect in production) are governed by their own policies.
